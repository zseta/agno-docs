Every team battles context sprawl. Knowledge ends up scattered across chat, drives, repos, and wikis, and no one person holds it all in their head.

**Scout is the teammate who does.**

Scout is a context agent: an agent that queries information sources to assemble context on demand. It follows the "navigation over search" pattern that makes coding agents so effective. Instead of fetching chunks from a pre-built vector index, Scout navigates live sources the same way a human would.

## What you can ask it

```
What did engineering decide about the auth rewrite?
Find the PTO policy and tell me how it works for contractors.
What's in the q1-planning folder in Drive?
Search the repo for where context providers get registered.
What time is it in Tokyo right now?
Save Sarah Chen as a contact, she's the new head of platform at Acme.
```

Each of these routes to a different source. Slack for the auth decision. Drive for the planning folder. Filesystem for the repo. The CRM for saving Sarah. The web for follow-up research. Scout picks the right source automatically.

## Navigation over search

Most "knowledge agents" embed everything into a vector index, search the index at query time, return chunks. That works for a frozen corpus. It breaks when the corpus is alive.

Scout takes the opposite approach. Sources stay where they are. Scout navigates them at query time, the same way a human would: list folders, search for files, read full documents, follow links. No ingest, no embeddings, no stale index.

The trade-off: each query costs more LLM calls. The benefit: answers reflect the live state of every source, citations include exact paths, and adding a new source means registering one provider, not running an ingest pipeline.

## What Scout sees

Scout is built around the `ContextProvider` abstraction. Six providers ship in the box. Any MCP server plugs in. Custom providers are a four-method subclass.

| Provider | What it exposes | Default |
|----------|-----------------|---------|
| **Web** | `web_search`, `web_fetch` | Always on (Parallel's keyless MCP) |
| **Filesystem** | `list_files`, `search_files`, `search_content`, `read_file` | Always on (the Scout repo) |
| **CRM** | `query_crm`, `update_crm` | Always on (your contacts, projects, notes in local Postgres) |
| **Slack** | `search_workspace`, `get_channel_history`, `get_thread`, `list_users` | When `SLACK_BOT_TOKEN` is set |
| **Google Drive** | `search_files`, `list_files`, `read_file` | When `GOOGLE_SERVICE_ACCOUNT_FILE` is set |
| **MCP servers** | `query_mcp_<slug>` per server | Wired in `scout/contexts.py` |

Each provider exposes `query_<id>` (and optionally `update_<id>`) tools on Scout. The agent calls them like any other tool. The provider handles auth, navigation, and pagination on the way to the source.

## Why a separate identity for each source

Scout authenticates as **its own identity** against every source. A dedicated Slack bot, a dedicated Google service account, a dedicated MCP token. Not your credentials borrowed.

This matters because:

- **Audit trails stay clean.** Everything Scout does shows up as Scout, not as you.
- **Multiple humans can use the same Scout safely.** Access scopes belong to Scout, not to whichever human is logged in.
- **Permissions are explicit.** You decide what Scout can see by sharing folders / channels / repos with its identity. Nothing more.

The same principle holds when more sources land (Gmail, Calendar, GitHub). Scout will always have its own seat.

## How it's built

```
Scout (one agno.Agent)
  ├─ ContextProviders[]
  │    ├─ WebContextProvider     → query_web
  │    ├─ FilesystemContext...   → list_files / search_content / read_file
  │    ├─ DatabaseContext (CRM)  → query_crm / update_crm
  │    ├─ SlackContextProvider   → search_workspace / get_channel_history / ...
  │    ├─ GDriveContextProvider  → search_files / read_file
  │    └─ MCPContextProvider[]   → query_mcp_<slug> per server
  └─ list_contexts (built-in)
```

One agent, N providers, one LLM hop per turn. No router, no specialists. Each provider exposes namespaced tools. Scout picks which to call based on the question.

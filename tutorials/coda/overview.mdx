---
title: What is Coda?
description: "A code companion that lives in Slack and works alongside your team."
---

Coding agents help us write code faster. But most engineering work isn't writing code. It's understanding how things work, reviewing PRs, figuring out what broke and why, triaging the backlog, and deciding what's safe to change. That work happens in Slack and GitHub, not in an editor.

**Coda fills the gap.**

Coda lives in Slack as a teammate. It answers architecture questions about your codebase, reviews PRs, triages issues, plans work, and writes code in isolated worktrees that never touch main.

## What you can ask it

```
@Coda where is the webhook handler for Stripe events?
@Coda walk me through the signup flow
@Coda what changed on the feature/payments branch?
@Coda review PR #42
@Coda triage the open issues and label them
@Coda plan adding webhook support to the payments service
@Coda add rate limiting to /api/v1/users using the same pattern as /orders
@Coda what's the recommended way to handle rate limiting in Redis?
```

Each routes to a different specialist on the team. The Explorer reads code. The Coder writes in worktrees. The Planner breaks features into issues. The Researcher hits the web. The Triager labels and closes issues.

## A team, not an agent

| Member | Role | Tools |
|--------|------|-------|
| **Explorer** | Reads code, traces call chains | `CodingTools` (read-only), `GitTools`, `GithubTools`, `ReasoningTools` |
| **Coder** | Writes code in isolated worktrees, opens PRs | `CodingTools` (full), `GitTools` (worktree), `GithubTools`, `ReasoningTools` |
| **Planner** | Breaks features into ordered GitHub issues | `CodingTools` (read-only), `GitTools`, `GithubTools` (issues), `ReasoningTools` |
| **Researcher** | Hits the web for framework docs, library APIs, CVEs | `ParallelTools`, `ReasoningTools` |
| **Triager** | Labels, comments, closes issues based on the actual code | `CodingTools` (read-only), `GithubTools` (issues), `ReasoningTools` |
| **Leader** | Routes the request, posts to Slack | `SlackTools` |

The Leader picks who handles what. Each member has a charter and a focused tool set, so the right thing happens for the right kind of question.

## Three things make Coda useful

**It lives in Slack.** No new tool to adopt. The codebase comes to where the conversation already happens.

**It shows up on its own.** Three scheduled tasks ship by default:
- **Daily Digest** posts what merged yesterday, what PRs are waiting, what issues opened, what's gone stale.
- **Issue Triage** reviews open issues against the actual codebase, labels them, posts a summary.
- **Repo Sync** pulls latest changes every 5 minutes so Coda is always working with current code.

**It learns your conventions.** First time you ask for a new endpoint, generic patterns. After a few weeks of use, it follows your service layer conventions, your error handling, your naming style, your logging patterns. Powered by [Agno's Learning Machine](/learning/overview).

## What Coda doesn't do

- **Auto-merge.** Coda opens PRs. A human merges them.
- **Touch your main branch.** All code work happens in isolated git worktrees off main. If something goes wrong, the worktree gets removed on the next restart.
- **Send code outside your environment.** Coda runs in your infrastructure. The only external calls are to your configured LLM provider.

## How it's built

Built on [Agno](https://github.com/agno-agi/agno) and [AgentOS](/runtime/overview). Reads from a `repos.yaml` you configure. Authenticates to GitHub via a Fine-grained Personal Access Token scoped to the repos you grant.

The whole repo: [github.com/agno-agi/coda](https://github.com/agno-agi/coda).

## Try it

```bash
git clone https://github.com/agno-agi/coda && cd coda
cp example.env .env  # set OPENAI_API_KEY and GITHUB_ACCESS_TOKEN
# edit repos.yaml to point at your repo (or just use the agno repo)
docker compose up -d --build
```

Then chat with it via [os.agno.com](https://os.agno.com). Once that works, [connect Slack](/tutorials/coda/connect-slack) and Coda starts living where your team does.

AgentOS deploys like any Python service: a container, a Postgres database, a public hostname, and a few env vars. The fastest path is a **Starter**: a clean AgentOS repo wired for one deploy target, with the provisioning scripts already written.

## Pick a Starter

Nine Starters cover the major platforms. Every Starter shares the same core and swaps only the deploy layer. Switch platforms later by swapping the deploy scripts.

| Starter | Target | Status |
|---------|--------|--------|
| [agentos-railway](https://github.com/agno-agi/agentos-railway) | Railway | Reference |
| [agentos-docker](https://github.com/agno-agi/agentos-docker) | Self-hosted Docker | Verified end-to-end |
| [agentos-fly](https://github.com/agno-agi/agentos-fly) | Fly.io | Verified end-to-end |
| [agentos-gcp](https://github.com/agno-agi/agentos-gcp) | Google Cloud Run | Verified end-to-end |
| [agentos-aws](https://github.com/agno-agi/agentos-aws) | AWS ECS Express | Verified end-to-end |
| [agentos-helm](https://github.com/agno-agi/agentos-helm) | Kubernetes (Helm) | Verified on kind; cloud verification pending |
| [agentos-azure](https://github.com/agno-agi/agentos-azure) | Azure Container Apps | Cloud verification pending |
| [agentos-render](https://github.com/agno-agi/agentos-render) | Render | Cloud verification pending |
| [agentos-modal](https://github.com/agno-agi/agentos-modal) | Modal + Neon Postgres | Cloud verification pending |

Starters handle deployment. Apps (Dash, Coda, Scout, Context) are complete, forkable agent products. Both live in [Templates](/deploy/templates).

## The reference flow

Every Starter follows the same shape as [agentos-railway](https://github.com/agno-agi/agentos-railway): run locally with Docker Compose, then deploy with the platform scripts.

```bash
git clone https://github.com/agno-agi/agentos-railway.git agentos
cd agentos
cp example.env .env    # set OPENAI_API_KEY
docker compose up -d --build
curl http://localhost:8000/health
```

Compose runs two services: the app and `agnohq/pgvector:18`, Postgres 18 with the pgvector extension preinstalled for knowledge embeddings.

Deploy from the same repo:

```bash
railway login
./scripts/railway/up.sh        # provision Postgres + app, assign a public domain
./scripts/railway/env-sync.sh  # sync .env.production to the service
./scripts/railway/redeploy.sh  # push code updates
./scripts/railway/down.sh      # tear everything down
```

`up.sh` provisions Postgres and the app service on a private network, assigns the public domain, writes it to `AGENTOS_URL`, and pauses for a `JWT_VERIFICATION_KEY` so production auth doesn't block the first deploy. The other Starters swap in their platform's deploy layer: `scripts/fly/`, `scripts/gcp/`, `scripts/aws/`, `charts/` for Helm, `compose.prod.yaml` for self-hosted Docker.

Walkthrough: [Railway deploy](/deploy/templates/railway/deploy).

## Bring your own infra

Not using a Starter? AgentOS is a FastAPI app. Any container platform works.

```python
from agno.os import AgentOS

agent_os = AgentOS(agents=[agent], db=db, tracing=True)
app = agent_os.get_app()

if __name__ == "__main__":
    agent_os.serve(app="my_app:app", reload=False)
```

| Resource | Why |
|----------|-----|
| **Container host** | Runs the FastAPI process |
| **PostgreSQL** | Sessions, memory, knowledge, traces, schedules |
| **Public hostname** | Required for Slack, Telegram, WhatsApp interfaces |
| **HTTPS** | Required for every webhook interface; terminate at your load balancer or reverse proxy |
| **Env vars** | At minimum `OPENAI_API_KEY` and `JWT_VERIFICATION_KEY` (in prod) |

AgentOS handles queues, worker pools, the scheduler, and JWT auth in-process. No separate worker fleet, no separate auth server, no separate cron container. Point health checks at `/health`; AgentOS responds with `"status": "ok"` when the app is ready.

See [Run Your AgentOS](/agent-os/run-your-os) for the minimal setup and [Bring Your Own FastAPI App](/agent-os/custom-fastapi/overview) for mounting AgentOS into an existing app.

## Scaling

AgentOS is stateless. State lives in `db`. Scale horizontally:

| Concern | Solution |
|---------|----------|
| Throughput | Add app replicas behind a load balancer |
| LLM rate limits | Use a queue or rate limiter in front of the model client |
| Long-running runs | Use `background=true` on the run endpoint, then poll for completion (see [Agent API](/features/api)) |
| Side effects without blocking the response | Background hooks with `run_in_background=True` |
| Schedule fan-out | The scheduler runs on a single replica's lifespan; for HA, use leader election or pin scheduling to one replica |
| Trace volume | Use a separate `trace_db` to keep the primary lean (see [Observability](/features/observability)) |

For the leader-election pattern with multiple replicas, see [Scheduler HA](/agent-os/scheduler/overview).

## Production checklist

**Auth and secrets**
- `RUNTIME_ENV=prd` enables JWT auth
- `JWT_VERIFICATION_KEY` set (see [Security & Auth](/features/security-and-auth))
- `OPENAI_API_KEY` and other model keys in a secret manager, not in source

**Infrastructure**
- Postgres has a persistent volume or managed backup
- HTTPS terminating at your load balancer or reverse proxy
- Health check pointed at `/health`

**Operational**
- Tracing on (`tracing=True`) so you can debug bad runs
- At least one [interface](/features/interfaces) wired up
- [Pre-hooks](/features/human-approval) for PII or injection guarding if you handle untrusted input
- `requires_confirmation=True` on irreversible tools

## Updating your deployment

| Change | How to deploy |
|--------|---------------|
| Code | `git push` if CI auto-deploys, or your Starter's redeploy script (`./scripts/railway/redeploy.sh`) |
| Env vars | The env sync script (`./scripts/railway/env-sync.sh`); Railway auto-redeploys when env values change |
| Database schema | Stock AgentOS tables need no migration tool: AgentOS manages them, and schema changes are additive and forward-compatible. Migrate your own application tables with Alembic, raw SQL, or dbt |

Scout is a working product agent you can clone and run. It navigates Slack, Google Drive, the web, a codebase, and any MCP server to answer questions about a company, and files what it learns into its own wiki and CRM. It is every concept in this section, shipped.

```bash
git clone https://github.com/agno-agi/scout && cd scout
cp example.env .env   # set OPENAI_API_KEY
docker compose up -d --build
curl http://localhost:8000/health   # {"status":"ok"}
```

## Where each concept shows up

| Concept | In Scout |
|---------|----------|
| [Serve as an API](/use-cases/product-agents/serve-as-an-api) | Ships as an AgentOS service behind Docker Compose, health check at `/health` |
| [Connecting your data](/use-cases/product-agents/connecting-your-data) | Five context providers always on (web, workspace, CRM, knowledge, voice), three opt-in (Slack, Drive, MCP) |
| [Sessions and memory](/use-cases/product-agents/sessions-and-memory) | The CRM and knowledge wiki are the closed loop: what Scout learns it files for next time |
| [Interfaces](/use-cases/product-agents/interfaces) | Slack loads when its env vars are set, using the conditional-registration pattern |

## The closed loop

Scout shows why a writable provider matters. As it works, it files what it learns:

> "Josh from Anthropic shared a new RLM paper this week."

Scout adds Josh to `scout_contacts`, files the paper as a linked note, and writes a wiki page summarizing it. The next question about Josh is faster because the answer is now in the CRM, not buried in a Slack thread. Put a daily cron on `scout_followups` and pending items surface every morning without anyone asking.

## Adapt it

Scout is a starting point, not a product. Point the context providers at your sources by editing [`scout/contexts.py`](https://github.com/agno-agi/scout/blob/main/scout/contexts.py): your S3 buckets, your Drive folders, your MCP servers. The goal is one agent that does one useful thing on real data, then iterate from real usage.

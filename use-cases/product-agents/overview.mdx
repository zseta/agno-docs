Software is becoming agentic and many parts of your product will become agent-driven in the near future. Most products start with a chat copilot, then add more agent-driven surfaces: smart empty states, inline workflow recommendations, personalized help notes. Agno serves every one of them from one backend that shares sessions, memory, knowledge, and auth.

The product feels coherent because there is one system powering it all.

```python
from agno.agent import Agent
from agno.db.postgres import PostgresDb
from agno.models.openai import OpenAIResponses
from agno.os import AgentOS

db = PostgresDb(db_url="postgresql+psycopg://ai:ai@localhost:5532/ai")

agent = Agent(
    name="Product Copilot",
    model=OpenAIResponses(id="gpt-5.5"),
    db=db,
    add_history_to_context=True,
    num_history_runs=5,
    update_memory_on_run=True,
)

agent_os = AgentOS(agents=[agent])
app = agent_os.get_app()

if __name__ == "__main__":
    agent_os.serve(app="copilot:app", port=7777)
```

That is a production agent for your product. Persistent multi-turn history, per-user memory, and an HTTP API with streaming, sessions, and auth. You own the product surface. Agno owns the runtime.

## Auth and tenancy are wired in

Turn on JWT auth and identity is read from the bearer token, so no surface has to pass `user_id` or `session_id` in the request body.

```python
from agno.os import AgentOS

agent_os = AgentOS(agents=[agent], authorization=True)
# Every endpoint requires a valid JWT. user_id, session_id, and
# scope claims are read from the token and applied before any
# agent code runs.
```

Sessions, memory, and knowledge scope per user, per organization, and per app. Multi-tenant isolation is the default. You do not write the glue code.

## Why Agno fits this shape

| Reason | What it means for your product |
|--------|--------------------------------|
| Sessions are native | A `session_id` ties every turn, its history, and its state together. You do not build the conversation store. |
| Memory is native | The agent learns user facts and recalls them on any surface. No retrieval glue to write. |
| The agent is an API | `AgentOS` turns the agent into a FastAPI service. Any surface calls `POST /agents/{id}/runs`. |
| One agent, many surfaces | The same agent answers a chat panel, an inline action, and a Slack mention. Memory follows the user across all of them. |
| Multi-tenant by default | Sessions, memory, and knowledge scope per user, per organization, and per app. |

## The mental model

A product surface does not embed an agent. It calls one.

| Surface | How it reaches the agent |
|---------|--------------------------|
| In-app copilot panel | `POST /agents/{id}/runs` with the user's `user_id` and a per-thread `session_id` |
| Inline action ("summarize this", "draft a reply") | The same endpoint, a short-lived `session_id` |
| Slack / Telegram / WhatsApp | An interface adapter maps the channel onto runs and sessions |
| Backend event (webhook, cron) | A custom FastAPI route calling `agent.arun(...)` |

Every surface shares the same `(user_id, session_id)` state. A user the agent met in the chat panel on Monday is already known to the inline action on Wednesday.

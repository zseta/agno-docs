Product agents can support chat, inline actions, personalized help, and proactive workflows.

Many products start with a chat copilot, then add smart empty states, inline workflow recommendations, personalized help notes, and proactive nudges. Each agent can own a specific job.

Agno can run them in one AgentOS deployment. Clients can share sessions, memory, and knowledge when they use consistent identities and storage.

```python
from agno.agent import Agent
from agno.db.postgres import PostgresDb
from agno.models.openai import OpenAIResponses
from agno.os import AgentOS

db = PostgresDb(db_url="postgresql+psycopg://ai:ai@localhost:5532/ai")

agent = Agent(
    name="Product Copilot",
    model=OpenAIResponses(id="gpt-5.5"),
    db=db,
    add_history_to_context=True,
    num_history_runs=5,
    enable_agentic_memory=True,
)

agent_os = AgentOS(agents=[agent])
app = agent_os.get_app()

if __name__ == "__main__":
    agent_os.serve(app="copilot:app", port=7777)
```

That is the start of a product copilot with multi-turn history, per-user memory, and an HTTP API with streaming and sessions.

## Configure authentication and user isolation

AgentOS supports JWT authentication and scope checks. Configure a JWT verification source, then enable authorization. Set `user_isolation=True` to scope non-admin session, memory, trace, and run access to the JWT subject.

```python
from agno.os import AgentOS
from agno.os.config import AuthorizationConfig

# Set JWT_VERIFICATION_KEY or JWT_JWKS_FILE before creating the app.
agent_os = AgentOS(
    agents=[agent],
    authorization=True,
    authorization_config=AuthorizationConfig(user_isolation=True),
)
# Every endpoint except public routes like /health and /docs now
# requires a valid JWT. user_id, session_id, and scopes are read
# from the token before any agent code runs.
```

## The mental model

The agent runs in your AgentOS as a service. Every product surface is a client that calls it over HTTP.

| Surface | How it reaches the agent |
|---------|--------------------------|
| In-app copilot panel | `POST /agents/{id}/runs` scoped to `user_id` and a per-thread `session_id` |
| Inline action ("summarize this", "draft a reply") | The same endpoint, a short-lived `session_id` |
| Slack / Telegram / WhatsApp | An interface adapter maps the channel onto runs and sessions |
| Backend event (webhook, cron) | A custom FastAPI route calling `agent.arun(...)` |

Calls that use the same agent, `user_id`, and `session_id` share session history. Agentic memory is keyed by `user_id`, so clients must map identities consistently across surfaces.

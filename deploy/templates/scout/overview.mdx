**Scout is a company intelligence agent that gets smarter as it explores your systems.**

Most AI assistants wait for you to upload documents or paste context. Scout navigates to the source. It connects to the web, Slack, Google Drive, and any MCP server, assembles context on demand, and builds its own wiki and CRM as it learns about your company. Check out the [repo](https://github.com/agno-agi/scout) for implementation notes.

## How It Works

Scout is a single agent with multiple **context providers**. Each provider exposes two natural-language tools: `query_<source>` for reads and `update_<source>` for writes when the source supports it. A sub-agent behind each provider owns the source's quirks. Scout sees `query_slack`; the sub-agent behind it knows to paginate by cursor and expand threads with `conversations.replies`.

| Provider | Active When | Tools |
|----------|-------------|-------|
| **Web** | Always on | `query_web`. Uses the Parallel SDK when `PARALLEL_API_KEY` is set, otherwise Parallel's free MCP server. |
| **Workspace** | Always on | `query_workspace`. Rooted at the Scout repo, so Scout can answer questions about its own codebase. |
| **CRM** | Always on | `query_crm`, `update_crm`. Contacts, projects, notes, follow-ups. |
| **Knowledge wiki** | Always on | `query_knowledge`, `update_knowledge`. Scout's prose memory. |
| **Voice wiki** | Always on | `query_voice`. A code-managed style guide for emails, Slack, X, and long-form writing. |
| **Slack** | `SLACK_BOT_TOKEN` | `query_slack`. Read-only access to messages, channel history, threads, and users. |
| **Google Drive** | `GOOGLE_SERVICE_ACCOUNT_FILE` | `query_gdrive`. Read-only access to files, folders, and contents. |
| **MCP** | Registered in `scout/contexts.py` | One `query_mcp_<slug>` per server. |

Provider setup guides:

- [Slack](/tutorials/scout/connect-slack)
- [Google Drive](/tutorials/scout/connect-google-drive)
- [MCP servers](/tutorials/scout/connect-mcp-servers)

Scout treats your company's sources the way coding agents treat a codebase: it navigates. Search the channel, open the doc, follow the link, expand the thread. Context is assembled at question time from the live source.

### Self-Building Knowledge

As Scout works, it maintains two internal systems. See [Knowledge wiki and CRM](/tutorials/scout/knowledge-and-crm) for details.

| System | Purpose |
|--------|---------|
| **Wiki** | Scout's prose memory: pages about your company, projects, and runbooks |
| **CRM** | People and relationships: contacts, projects, notes, follow-ups |

Both start empty and grow with use. When you mention that Josh from Anthropic shared a new RLM paper, Scout adds Josh to the CRM, parses the paper into the wiki, and links the two.

## Run Locally

See the [Setup guide](/tutorials/scout/setup) for detailed instructions. You need [Docker Desktop](https://docs.docker.com/desktop/) installed and running.

```bash
git clone https://github.com/agno-agi/scout && cd scout

cp example.env .env
# set OPENAI_API_KEY in .env

docker compose up -d --build
```

Scout is now running at `http://localhost:8000`.

### Connect to the AgentOS UI

1. Open [os.agno.com](https://os.agno.com) and log in.
2. Click **Connect OS**, choose **Local**, and enter `http://localhost:8000`.
3. Try the pre-configured prompts.

## Deploy to Railway

Scout runs on any cloud provider; the repo ships scripts for Railway. See [Deploy to Railway](/tutorials/scout/deploy-to-railway) for JWT setup and the Git-backed wiki.

```bash
cp .env .env.production

railway login
./scripts/railway/up.sh
```

Edit `.env.production` where production values differ from local. The Railway scripts read `.env.production` first and fall back to `.env`. The `up.sh` script provisions PostgreSQL and the app service.

Expect the first deploy to fail: production Scout refuses to serve traffic without a `JWT_VERIFICATION_KEY`. To fix it:

1. Open [os.agno.com](https://os.agno.com), click **Connect OS** → **Live**, and enter your Railway domain.
2. Enable **Token Based Authorization**.
3. Paste the public key into `.env.production` as `JWT_VERIFICATION_KEY` (the full PEM block, no surrounding quotes).
4. Sync the env. Railway auto-deploys when values change.

```bash
./scripts/railway/env.sh       # sync .env.production to Railway
./scripts/railway/redeploy.sh  # push code updates after up.sh
```

## Connect to Slack

Scout is designed to live in Slack as a teammate. Each Slack thread becomes a session with its own context, so follow-ups in the same thread carry forward.

1. Get a public URL Slack can reach: `ngrok http 8000` locally, or your Railway domain in production.
2. Create the Slack app from the manifest in [`docs/SLACK_CONNECT.md`](https://github.com/agno-agi/scout/blob/main/docs/SLACK_CONNECT.md).
3. Set `SLACK_BOT_TOKEN` and `SLACK_SIGNING_SECRET` in `.env`.
4. Restart Scout.

See [Slack setup](/tutorials/scout/connect-slack) for scopes, events, and the production Request URL.

## Run Evals

Scout ships three eval tiers. PostgreSQL must be running for every tier: `docker compose up -d scout-db`.

| Tier | Command | What It Catches |
|------|---------|-----------------|
| **Wiring** | `python -m evals wiring` | Tool shape drift and missing schema guards. Code-level invariants, no LLM. |
| **Behavioral** | `python -m evals` | Wrong tool choices, missing response content, forbidden tools firing. |
| **Judges** | `python -m evals judges` | Answer quality, LLM-scored. |

Run a single case with `python -m evals --case <id>`. See [`docs/EVALS.md`](https://github.com/agno-agi/scout/blob/main/docs/EVALS.md) for the full picture.

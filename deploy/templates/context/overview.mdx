**@context is a self-hosted context manager: a private CRM and knowledge base with one agent in front.**

It plugs into Claude, ChatGPT, Claude Code, Codex, and Cursor over MCP and gives them a single source of context about your work. Anyone can write to it. Only you can read it or act through it.

@context is an [App](/deploy/introduction): a complete agent product you fork, run on your own infrastructure, and adapt. See the [repo](https://github.com/agno-agi/context) for the source.

## How It Works

@context is one Agno agent running on the AgentOS runtime. AgentOS verifies the caller's identity on every request, from the JWT or the Slack event, and assigns tools by role:

1. **Owner mode.** You get every tool: capture context, retrieve it (*"give me a rundown of my day"*), and prepare it (*"process today"*).
2. **Guest mode.** Teammates and their agents get one tool, `submit_update`, which drops an update in your queue. You get briefed in your next rundown.

### Five Jobs

1. **Maintain a CRM.** Say *"met Kyle from Agno, wants a partnership, follow up next week"* and @context stores a contact, a note, and a dated reminder without you picking forms or fields.
2. **Maintain a knowledge base.** @context writes product specs, parses customer interview notes, manages project briefs, and runs deep research, then keeps it all organized in one place.
3. **Run your day, plan your week, prep ahead.** Repeatable playbooks handle the routine, on demand or on a schedule.
4. **Represent you.** A teammate types *"@your-context my claude fixed the auth bug"* and it lands in your queue. It works outbound too: @context can message people and channels on Slack on your behalf, and @-mention a teammate's @context to drop an update in their queue.
5. **Draft and schedule.** Connect Gmail and Calendar, and @context drafts follow-ups straight into Gmail for you to send and routes calendar changes through your approvals queue.

The built-in playbooks:

| Playbook | Ask | What you get |
|----------|-----|--------------|
| **Rundown** | "What's on today?" | One prioritized brief: queue updates, due reminders, today's meetings, missed emails, and the Slack threads worth a look |
| **Week plan** | "What's my week?" | Priorities for the week, delivered to your Slack DMs on Sunday evening |
| **Prep** | "Prep for my 2pm with Kyle" | A pre-meeting brief: who they are, past notes and threads, open items, and public background for contacts you don't know yet |

Playbooks live in `skills/`, one `SKILL.md` per folder, alongside `process-today`, `research`, and `knowledge-review`. Customize them and add your own.

### Context Sources

Each source is a sub-agent behind at most two tools: `query_<id>` to read and `update_<id>` to write.

| Source | Access | Enabled by |
|--------|--------|------------|
| `crm` | Read and write | Always on |
| `knowledge` | Read and write | Always on |
| `workspace` | Read | Always on |
| `web` | Read | Always on |
| `slack` | Read and write | `SLACK_BOT_TOKEN` |
| `gmail` | Read and write; writes are drafts only | `GOOGLE_*` credentials |
| `calendar` | Read and write; writes pause for approval | `GOOGLE_*` credentials |

The `crm` source is the structured store: contacts, projects, meetings, reminders, and notes in the `crm` Postgres schema, with every row scoped to your `user_id`. The `knowledge` source holds specs and research notes, on the filesystem locally and backed by a Git repo in production.

### Security

@context holds a lot of sensitive information, so the boundaries are structural. The toolset is assembled in code, from a verified identity, before the model runs. A guest never holds a read tool into your data; to everyone else, @context is a polite notetaker that only captures. Each caller gets their own user memory, kept entirely separate from yours.

External actions are double gated. `update_calendar` pauses for your explicit approval before it runs. `update_gmail` writes drafts only, so the follow-up waits in your Gmail drafts for you to review and send.

Everything runs locally or in your own cloud. The CRM, the knowledge base, and the inbox live in your own database. See [docs/SECURITY.md](https://github.com/agno-agi/context/blob/main/docs/SECURITY.md).

## Run Locally

> **Prerequisites:** [Docker](https://www.docker.com/get-started/) installed and running. An [OpenAI API key](https://platform.openai.com).

```bash
git clone https://github.com/agno-agi/context.git && cd context

cp example.env .env
# Edit .env: set OPENAI_API_KEY, and set OWNER_ID to the email you sign in to os.agno.com with

docker compose up -d --build
```

Confirm it is live at [localhost:8000/docs](http://localhost:8000/docs).

`OWNER_ID` decides who gets the owner surface. Set it to your email; requests from any other identity get the capture-only guest surface.

### Connect the AgentOS UI

1. Open [os.agno.com](https://os.agno.com) and sign in with the email you set as `OWNER_ID`.
2. Click **Connect OS** → **Local**, enter `http://localhost:8000`, and name it **Local Context**.
3. Open the chat under Context and try one of the quick prompts.

Use the AgentOS UI to chat with @context, view sessions, and approve actions.

### Connect MCP Clients

The main way to use @context day to day is from an MCP client. The server runs at `http://localhost:8000/mcp` and is owner-only.

```bash
python scripts/connect.py
```

The script finds Claude Code, Codex, Claude Desktop, and Cursor, and registers @context with each. Use `--dry-run` to preview and `--remove` to undo.

To register the CLI clients by hand:

```bash
claude mcp add -s user --transport http context http://localhost:8000/mcp
codex mcp add --url http://localhost:8000/mcp context
```

Claude Desktop and Cursor take config-file entries. See [docs/MCP.md](https://github.com/agno-agi/context/blob/main/docs/MCP.md) for both.

## Deploy to Railway

@context runs anywhere that runs a Docker container. For a quick deployment, `scripts/railway/up.sh` runs @context and PostgreSQL on the same private network and creates the public domain you connect to in the AgentOS UI.

> **Prerequisites:** [Railway CLI](https://docs.railway.com/cli#installing-the-cli) installed and `railway login` completed.

<Steps>
  <Step title="Create a production env">
```bash
cp .env .env.production
```

The deploy scripts read `.env.production` first and fall back to `.env`.
  </Step>

  <Step title="Deploy">
```bash
./scripts/railway/up.sh
```

The script pauses while you mint a JWT verification key.
  </Step>

  <Step title="Mint your JWT key">
Token-Based Authorization is on by default. Without a `JWT_VERIFICATION_KEY` in `.env.production`, the AgentOS will not serve traffic. That is the safe default for an agent that holds your work context. You can also [issue and verify your own JWT](/agent-os/security/authorization/self-hosted).

1. Open [os.agno.com](https://os.agno.com), click **Connect OS** → **Live**, and enter the domain the script printed.
2. Turn on **Token-Based Authorization** and click **Connect**.
3. Copy the public key into `.env.production` as `JWT_VERIFICATION_KEY`.
4. Back in the terminal, press Enter. The script reads the key and deploys the service.
  </Step>

  <Step title="Verify">
```bash
railway logs --service agent-os
```

Open `https://<your-domain>/docs` to confirm the API is serving.
  </Step>
</Steps>

Env changes sync with `./scripts/railway/env-sync.sh`. Code changes redeploy with `./scripts/railway/redeploy.sh`, or connect the repo in the Railway dashboard for auto-deploy on merge.

### Point MCP Clients at Production

The deployed server is JWT-gated, so your MCP clients need a bearer token. One command mints one and rewires everything:

```bash
./scripts/venv_setup.sh          # once; minting needs pyjwt + cryptography
source .venv/bin/activate
./scripts/setup_context.sh
```

`setup_context.sh` mints a self-issued token (the private key stays local in gitignored `secrets/`), pushes the public key to Railway, redeploys, and points Claude Code, Codex, Claude Desktop, and Cursor at the production endpoint. Re-run it any time to rotate the token; add `--no-redeploy` to skip the redeploy. @context trusts your key alongside the os.agno.com one, so the AgentOS UI keeps working.

### Back the Knowledge Base with Git

Local runs store the knowledge base in the gitignored `knowledge/` folder. In production, back it with a private Git repo: create one, mint a fine-grained token with read and write access to its contents, and add both to `.env.production`:

```bash
KNOWLEDGE_REPO_URL=https://github.com/you/your-context.git
KNOWLEDGE_GITHUB_TOKEN=ghp_...
```

Then sync with `./scripts/railway/env-sync.sh`. See [docs/KNOWLEDGE.md](https://github.com/agno-agi/context/blob/main/docs/KNOWLEDGE.md).

## Connect to Slack

Slack is where @context comes alive: teammates @-mention it to leave you updates, you DM it for private conversations, and the daily rundown and weekly plan land in your DMs on schedule.

1. Create a Slack app from the manifest in [docs/SLACK.md](https://github.com/agno-agi/context/blob/main/docs/SLACK.md).
2. Copy the Bot User OAuth Token and Signing Secret.
3. Set `SLACK_BOT_TOKEN` and `SLACK_SIGNING_SECRET` in `.env` or `.env.production`.
4. Restart the application.

Local runs need a public URL for Slack's callbacks; use an ngrok tunnel. After deploying, repoint the app's `/slack/events` and `/slack/interactions` request URLs at your Railway domain.

## Connect Gmail and Calendar

Connect Gmail and Calendar to ground the rundown and meeting prep in your real inbox and calendar. Set `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, and `GOOGLE_PROJECT_ID`, then mint tokens with `python scripts/google_mint_tokens.py`. See [docs/GOOGLE.md](https://github.com/agno-agi/context/blob/main/docs/GOOGLE.md).

## Run Evals

The eval suite ([`evals/`](https://github.com/agno-agi/context/tree/main/evals)) is a regression test for the security model. Deterministic gates prove the guest toolset stays capture-only and the MCP server stays owner-only; an adversarial guest arc tries to break the boundary.

```bash
python -m evals                # run the full suite
python -m evals -v             # stream the full agent run
python -m evals --case <name>  # one case
```

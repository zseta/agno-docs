[`agno-agi/agentos-helm`](https://github.com/agno-agi/agentos-helm) is the Kubernetes [Starter](/deploy/introduction): it runs locally on Docker and deploys AgentOS with PostgreSQL (pgvector) to any Kubernetes cluster using the Helm chart in `charts/agentos` and one script. Cloud-managed clusters (EKS, GKE, AKS) and your own both work.

It ships with three agents:

| Agent | Role |
|-------|------|
| **Agent Builder** | Creates agents, teams, and workflows through the [AgentOS Studio](/agent-os/studio/introduction). |
| **Platform Manager** | Read-only ops: eval history, deployment checks, schedules, codebase questions. |
| **WebSearch** | Searches the web with Parallel. The simplest agent to copy. |

## Set up with a coding agent

The fastest way to a running platform. Copy the prompt into Claude Code, Cursor, or Codex; the agent clones the repo, configures credentials, starts Docker, proves the MCP endpoint end to end, and connects the frontends you pick.

```text expandable
Help me set up an AgentOS on this machine. Work step by step and verify each step before moving to the next. When a step needs me (an API key, a Docker install, a sign-in), stop, tell me exactly what to do, and wait for my input. Never read or print secrets.

1. Clone https://github.com/agno-agi/agentos-helm.git and cd in. Then read AGENTS.md end to end — it is the source of truth for how this platform works and answers most questions you'll hit along the way.
2. Run `cp example.env .env`, open .env in my favorite editor, and ask me to set the OPENAI_API_KEY.
3. Confirm docker is installed, running and `docker info` succeeds. If Docker is missing, ask me to install Docker Desktop and wait until it's running.
4. Start the platform with `docker compose up -d --build`, then poll http://localhost:8000/docs until it returns 200 (first build takes a few minutes). If it never comes up, read `docker compose logs agentos-api` and fix what you find.
5. Prove it end to end with ./scripts/mcp_check.sh — it should print "MCP OK" and a real agent answer. Show me that answer: it's my platform talking.
6. Ask me which frontends I want connected, then set up the ones I pick:
   - Coding agents (including you): run `uvx agno connect` — it registers http://localhost:8000/mcp in Claude Code, Claude Desktop, Codex, and Cursor, and verifies with a real handshake. Use the default user scope, not --project (that would write a token into the repo).
   - The AgentOS web UI: walk me through os.agno.com → Connect OS → http://localhost:8000, named "Local AgentOS".
   - Claude and ChatGPT apps (web or desktop): their sessions run in the cloud and can't reach localhost, so work with me on deploying to production first using ./scripts/k8s/up.sh (needs kubectl pointed at a cluster, helm, and an image the cluster can pull; the script pauses while I mint a JWT key at os.agno.com). Then I add https://<my-agentos-domain>/mcp as a custom connector in the chat app's connector settings.
7. Finish with a short summary of what's running and where, plus a few first prompts to try — start with asking Agent Builder to "Build an agent that tracks AI news and writes a daily brief".
```

Prefer to drive yourself? Follow the manual steps below.

## Manual setup

> **Prerequisites:** [Docker](https://www.docker.com/get-started/) installed and running. An [OpenAI API key](https://platform.openai.com).

<Steps>
  <Step title="Clone and configure">
```bash
git clone https://github.com/agno-agi/agentos-helm.git agentos
cd agentos

cp example.env .env
```

Edit `.env` and set `OPENAI_API_KEY`.
  </Step>

  <Step title="Start the platform">
```bash
docker compose up -d --build
```

The first build takes a few minutes. Confirm the API at [localhost:8000/docs](http://localhost:8000/docs).
  </Step>

  <Step title="Verify end to end">
```bash
./scripts/mcp_check.sh
```

Prints `MCP OK` with the tool count and a real agent answer through the MCP endpoint.
  </Step>

  <Step title="Connect the AgentOS UI">
1. Open [os.agno.com](https://os.agno.com) and sign in.
2. Click **Connect OS**, enter `http://localhost:8000`, and name it **Local AgentOS**.
  </Step>

  <Step title="Build your first agent">
Chat with **Agent Builder**: "Build an agent that tracks AI news and writes a daily brief". Then ask **Platform Manager**: "How healthy is the platform?"
  </Step>
</Steps>

<Check>Your AgentOS is running locally.</Check>

## Connect your frontends

| Frontend | How |
|----------|-----|
| Coding agents (Claude Code, Codex, Cursor) | `uvx agno connect` registers `http://localhost:8000/mcp` and verifies with a real handshake. |
| AgentOS UI | [os.agno.com](https://os.agno.com) → **Connect OS** → `http://localhost:8000`. |
| Claude and ChatGPT apps | Hosted sessions can't reach localhost. Deploy to production first, then add `https://<your-domain>/mcp` as a custom connector. |
| Slack | Set `SLACK_BOT_TOKEN` and `SLACK_SIGNING_SECRET`. See [Slack setup](/deploy/interfaces/slack/overview). |
| Your product | REST API with 80+ endpoints. Browse them at `/docs`. |

## Deploy to production

> **Prerequisites:** [kubectl](https://kubernetes.io/docs/tasks/tools/) pointed at your cluster, [Helm](https://helm.sh/docs/intro/install/) 3+, and a container registry the cluster can pull from.

<Steps>
  <Step title="Create a production env">
```bash
cp .env .env.production
```

Edit `.env.production` with production values: a different OpenAI key, production-only credentials, a different Slack workspace.
  </Step>

  <Step title="Choose your image">
The chart defaults to the official [`agnohq/agentos`](https://hub.docker.com/r/agnohq/agentos) image: the reference platform exactly as in this repo, tagged `latest` plus `agno-<pin>` for exact runtimes. As soon as you customize anything (a new agent, edited instructions), build and push your own:

```bash
docker build -t <registry>/agentos:v1 .
docker push <registry>/agentos:v1
```
  </Step>

  <Step title="Deploy">
```bash
./scripts/k8s/up.sh                                                    # official image
IMAGE_REPOSITORY=<registry>/agentos IMAGE_TAG=v1 ./scripts/k8s/up.sh   # your own build
```

Installs the chart into the `agentos` namespace of your current kubectl context. The script prints the context and asks before proceeding, then provisions the API deployment and in-cluster PostgreSQL with pgvector on a persistent volume.

The API runs one replica by design; the in-process scheduler double-fires crons on two replicas. The script also mints a `DB_PASS` and saves it to your env file. Keep it: the Postgres volume reads the password only on first initialization.

To publish behind your ingress controller, add `INGRESS_HOST=os.example.com` and optionally `INGRESS_CLASS=nginx`. `AGENTOS_URL` then points at that host; otherwise the scheduler uses the in-cluster service DNS, which works out of the box.
  </Step>

  <Step title="Mint your JWT key">
The script pauses for a `JWT_VERIFICATION_KEY`. Token-Based Authorization is on by default; without a key, the platform refuses to serve production traffic.

1. Open [os.agno.com](https://os.agno.com), click **Connect OS** → **Live**, enter your AgentOS URL (your ingress host, or a tunnel while testing), and name it **Live AgentOS**.
2. Go to **Settings** → **OS & Security** and turn on **Token-Based Authorization (JWT)**.
3. Copy the public key and paste the full PEM into the `up.sh` prompt. The script saves it to your env file and deploys.

If you skip the prompt, add the key to `.env.production` later and run `./scripts/k8s/env-sync.sh`.

<Note>Live AgentOS connections are a paid feature. Use code `PLATFORM30` for one month off.</Note>
  </Step>

  <Step title="Confirm it's live">
```bash
kubectl rollout status deployment/agentos -n agentos
kubectl logs deploy/agentos -n agentos -f
```

No ingress yet? Port-forward and open [localhost:8000/docs](http://localhost:8000/docs):

```bash
kubectl port-forward svc/agentos 8000:8000 -n agentos
```
  </Step>
</Steps>

<Check>Your AgentOS is live on your Kubernetes cluster.</Check>

### Production auth

Token-Based Authorization gives you three things:

1. **No public access.** The server rejects requests without a valid token.
2. **Per-request identity.** Middleware parses `user_id`, `session_id`, and custom claims from the token.
3. **Granular permissions.** User tokens run agents and read their own sessions. Admin tokens read everything.

The same middleware protects `/mcp`: clients authenticate with a bearer token, either a JWT minted at os.agno.com or a service-account PAT (`uvx agno connect` mints one automatically). To opt out inside a private VPC behind another auth layer, set `authorization=False` in `app/main.py` and redeploy.

### Redeploy after code changes

Build and push a new tag, then roll the release to it:

```bash
docker build -t <registry>/agentos:v2 . && docker push <registry>/agentos:v2
IMAGE_TAG=v2 ./scripts/k8s/redeploy.sh
```

Immutable tags keep rollbacks one `helm rollback` away. Running `./scripts/k8s/redeploy.sh` without `IMAGE_TAG` restarts the pods in place; that picks up a re-pushed tag only if the new image actually reached the cluster. Env changes still sync with `./scripts/k8s/env-sync.sh`.

### Dry run on kind

The whole production flow runs on a laptop-local [kind](https://kind.sigs.k8s.io) cluster:

```bash
kind create cluster --name agentos
docker build -t agentos:kind . && kind load docker-image agentos:kind --name agentos
printf 'RUNTIME_ENV=dev\n' > .env.production && grep '^OPENAI_API_KEY=' .env >> .env.production
IMAGE_REPOSITORY=agentos IMAGE_TAG=kind IMAGE_PULL_POLICY=Never ./scripts/k8s/up.sh
kubectl port-forward svc/agentos 8000:8000 -n agentos   # then open http://localhost:8000/docs
./scripts/k8s/down.sh --yes && kind delete cluster --name agentos && rm .env.production
```

`RUNTIME_ENV=dev` disables JWT, so nothing needs minting. Never sync a dev env file to a real cluster.

## Environment variables

The variables you touch during setup. Full table in the [reference](/deploy/templates/helm/reference#environment-variables).

| Variable | Required | Description |
|----------|----------|-------------|
| `OPENAI_API_KEY` | Yes | Models and embeddings. |
| `JWT_VERIFICATION_KEY` | Production | Public key from os.agno.com. The platform refuses production traffic without it. |
| `AGENTOS_URL` | No | Scheduler base URL. The chart resolves it: explicit value, then ingress URL, then in-cluster service DNS. |
| `SLACK_BOT_TOKEN` / `SLACK_SIGNING_SECRET` | No | Set both to enable the Slack interface. |
| `PARALLEL_API_KEY` | No | WebSearch uses the Parallel SDK when set, keyless MCP otherwise. |

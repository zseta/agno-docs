You have an agent, AgentOS running it, and a Postgres holding state. What's left to ship a product?

The honest answer is: less than you think. Most of the gap between "AgentOS works" and "users can pay" is product work, not infrastructure. This page is the punch list.

## Step 1: Pick the right starting point

Don't build AgentOS from scratch. Start from a template that's closest to what you want to ship.

| Template | Right when |
|----------|------------|
| [Scout](/tutorials/scout/setup) | You're building a context agent that pulls from external systems (Slack, Drive, MCP, custom) |
| [Dash](/tutorials/dash/setup) | You're building a data agent that answers questions from your database |
| [Coda](/tutorials/coda/setup) | You're building a code companion that lives in Slack |
| [Demo OS](/demo-os/overview) | You want a kitchen-sink reference with every feature wired up |
| Bare AgentOS | You want full control and don't mind wiring it yourself |

All four templates ship with Docker Compose, Railway scripts, Slack manifest, JWT setup, and a `.env.production` flow. You're 90% of the way to deploy on day one.

## Step 2: Replace the demo data with yours

Templates ship with synthetic data so the first run works. Replace it with your own:

| If you're building on... | Swap |
|--------------------------|------|
| Scout | Configure context providers in [`scout/contexts.py`](https://github.com/agno-agi/scout/blob/main/scout/contexts.py). Point at your S3 buckets, your Drive folders, your MCP servers. |
| Dash | Replace [`scripts/generate_data.py`](https://github.com/agno-agi/dash/blob/main/scripts/generate_data.py) with a loader for your dataset, or `pg_restore` directly into `public`. Then rewrite [`knowledge/`](https://github.com/agno-agi/dash/tree/main/knowledge) for your tables. |
| Coda | Edit [`repos.yaml`](https://github.com/agno-agi/coda/blob/main/repos.yaml) to point at your repos. Make sure your `GITHUB_ACCESS_TOKEN` has the right scopes. |
| Demo OS | Fork the agent that's closest to yours. Modify its instructions, knowledge, tools. Remove the agents you don't need. |

The goal is a single agent that does *one useful thing* on real data. Then iterate.

## Step 3: Pick your interfaces

Your users aren't going to curl your API. Pick the surfaces they actually use:

| Surface | When |
|---------|------|
| [Slack](/agent-os/interfaces/slack/introduction) | Internal tools, B2B teams, async work |
| [Telegram](/agent-os/interfaces/telegram/introduction) | Personal assistants, mobile-first, consumer |
| [WhatsApp](/agent-os/interfaces/whatsapp/introduction) | Customer support, mobile, international |
| [AG-UI](/agent-os/interfaces/ag-ui/introduction) | Custom web apps, embedded chat widgets |
| [A2A](/agent-os/interfaces/a2a/introduction) | Other agents calling yours |
| Custom HTTP | Webhooks, integrations, anything else |

Most products start with one and add more later. Start with the one your users are already on.

## Step 4: Wire auth

Production demands authentication. AgentOS does this with JWT and RBAC scopes. The flow:

1. Set `RUNTIME_ENV=prd` (or `authorization=True` directly).
2. Set `JWT_VERIFICATION_KEY` from [os.agno.com](https://os.agno.com).
3. Mint user tokens with appropriate scopes.

For most products, the [AgentOS control plane](https://os.agno.com) issues tokens for you. For custom auth (your own user database), implement the [authorization config](/runtime/security-and-auth) with your own verification keys.

Test it: hit any endpoint without a token, get 401. Hit it with a valid token, get the response.

## Step 5: Set up observability before you need it

Tracing is one flag (`tracing=True`). Turn it on day one. The first time a user reports a wrong answer, you'll have the trace tree to debug from.

| Concern | Action |
|---------|--------|
| Internal debugging | `tracing=True`, query `agno_traces` directly with SQL |
| Production monitoring | Send traces to [Langfuse, Arize, Langsmith, etc.](/runtime/observability#external-providers) |
| Eval-driven iteration | Run eval suites against your agent on a schedule, alert on regressions |
| Audit logs for compliance | Add `post_hooks=[audit_log]` with `run_in_background=True` |

The [observability page](/runtime/observability) has the SQL queries to start with.

## Step 6: Add HITL where it matters

If your agent does anything irreversible, gate it:

```python
@tool(requires_confirmation=True)
def send_customer_email(to: str, subject: str, body: str) -> str:
    return mail.send(to, subject, body)

@approval(type="required")
@tool
def issue_credit(customer_id: str, amount: float) -> str:
    return billing.credit(customer_id, amount)
```

Resist the urge to add approval everywhere. Approval friction kills adoption. Pick the few actions that genuinely need a human and gate just those.

## Step 7: Schedule the proactive work

Most useful agents do something on a schedule, not just on demand:

- A daily digest your team wakes up to
- Background processing of a queue
- Periodic health checks or summaries
- Weekly reports

Use the scheduler. Register schedules in your app's lifespan so they survive restarts:

```python
@asynccontextmanager
async def lifespan(app):
    register_schedule(name="daily_digest", cron="0 9 * * 1-5",
                      endpoint="/workflows/daily-digest/runs")
    yield
```

## Step 8: Deploy

Pick a host. Set env vars. Push.

| Host | Path |
|------|------|
| Railway | Use the template's `scripts/railway_*.sh` |
| AWS / GCP / Azure | Container service + managed Postgres + secret manager |
| Self-hosted | Docker Compose on a VM |

The [deploy page](/runtime/deploy) has the per-host details. The shape is the same: container, Postgres, secret store, public hostname, JWT key.

## Step 9: Iterate from real usage

The first version is wrong about things, and that's expected. The interesting question is *how fast can you find what's wrong*.

| Signal | Where |
|--------|-------|
| Wrong answers users complained about | `agno_sessions` joined with feedback table you build |
| Tools failing in production | `agno_spans` filtered by `status='error'` |
| Slow runs | `agno_spans` `end_time - start_time` per agent |
| Cost spikes | `agno_sessions.total_tokens` grouped by `agent_id` and day |
| Regression after a change | Run [evals](/evals/overview) before deploying |

The fastest improvement loop: a user complains, you find the run, you replay it locally, you fix the prompt or add a learning, you ship.

## Step 10: Keep the agent learning

The agents that get better over time are the ones with feedback loops. Three patterns the templates use:

| Pattern | Where |
|---------|-------|
| **LearningMachine** | Agent stores discovered patterns and retrieves them next time. See [Learning](/learning/overview). |
| **Knowledge updates** | When you find a wrong-answer pattern, add the right answer to knowledge. The next user gets it. |
| **Eval-gated deploys** | Every deploy runs an eval suite. Regressions block the merge. |

The agents that get *worse* over time are the ones nobody is watching. Don't let yours be one of them.

## What's left

You shipped. Now:

| Concern | Where |
|---------|-------|
| Pricing, billing, rate limits | Layer in front of AgentOS. Stripe, your own metering, your own limits. |
| Multi-tenant isolation | Per-tenant `db` instances or schema-per-tenant within one db |
| Compliance (SOC 2, HIPAA, etc.) | RBAC + audit logs + data residency choices in your storage backend |
| Custom UIs | AG-UI for the chat surface; build the rest as a normal web app calling AgentOS |

These are normal product concerns. AgentOS gives you the runtime, and the product is yours to ship.

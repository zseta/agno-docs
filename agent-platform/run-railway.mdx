The platform runs anywhere containers run. If your company has a set way of deploying software, follow that. For the lightest lift, the Starter ships Railway scripts:

| Script | What it does |
| ------ | ------------ |
| `./scripts/railway/up.sh` | First-time provisioning: project, database, app service, domain, deploy. |
| `./scripts/railway/env-sync.sh` | Push env vars from `.env.production` (or a file you pass) to the app service. |
| `./scripts/railway/redeploy.sh` | Redeploy the app from your machine. |
| `./scripts/railway/down.sh` | Delete the Railway project, including all data. |

## Prerequisites

- A [Railway](https://railway.app) account
- The [Railway CLI](https://docs.railway.com/cli#installing-the-cli) installed and authenticated (`railway login`)

## Deploy your agent platform to Railway

<Steps>
  <Step title="Configure your production environment">

    The deploy scripts read `.env.production` first, then fall back to `.env`. A separate file keeps local and production values apart: different OpenAI keys with different budgets, production-only credentials, a different Slack workspace.

    ```bash
    cp .env .env.production
    ```

    Edit `.env.production` with a production OpenAI key.

  </Step>

  <Step title="Provision and deploy">

    ```bash
    ./scripts/railway/up.sh
    ```

    The script:

    1. Creates a Railway project named `agentos-railway`.
    2. Provisions a PgVector database service with a persistent volume.
    3. Creates the `agent-os` app service and forwards the database connection vars. API keys are set quietly so they never print in the deploy log.
    4. Issues the public domain before deploying and sets `AGENTOS_URL` to it, so scheduled jobs can reach AgentOS. The value is written back to `.env.production`.
    5. Pauses for a JWT verification key (next step).
    6. Builds and deploys from the current directory.

    The domain takes a few minutes to start resolving after the deploy.

  </Step>

  <Step title="Mint your JWT verification key">

    Token-Based Authorization is on outside local dev. Without a `JWT_VERIFICATION_KEY` or `JWT_JWKS_FILE`, the app refuses to serve production traffic. That default keeps your data off the public web.

    Token-Based Auth gives you three things:

    - **No public access.** The server rejects requests without a valid token.
    - **Per-request identity.** Middleware parses the token and injects `user_id`, `session_id`, and custom claims into every request.
    - **Granular permissions.** A user token can run an agent and view its own sessions. An admin token can read everyone's sessions and test any agent.

    `up.sh` creates your domain first, then pauses for the key so the first deploy starts serving instead of failing on a missing key. While it waits:

    1. Open [os.agno.com](https://os.agno.com), click **Connect OS** → **Live**, enter your Railway domain, and name it **Live AgentOS**.
    2. Go to **Settings** → **OS & Security** and turn **Token-Based Authorization (JWT)** on. The UI generates a keypair, keeps the private key, and shows you the public key.
    3. Paste the full PEM block at the script's prompt. The script saves it to `.env.production` and sets it on Railway.

    <Note>
      Live AgentOS connections require a Pro subscription. Use coupon `PLATFORM30` for one month free.
    </Note>

    You can also bring your own keypair: sign tokens with the private key in your own auth service and put the matching public key in `JWT_VERIFICATION_KEY`. Or set `JWT_JWKS_FILE` if you mount a JWKS file in the image.

    If you skip the prompt, add the key to `.env.production` and run `./scripts/railway/env-sync.sh`. Quote the value so the multi-line PEM parses as one variable:

    ```bash
    JWT_VERIFICATION_KEY="-----BEGIN PUBLIC KEY-----
    MIIBIjANBgkq...
    -----END PUBLIC KEY-----"
    ```

  </Step>

  <Step title="Verify">

    ```bash
    railway logs --service agent-os
    ```

    Once you see successful requests, you're live. To push later env changes:

    ```bash
    ./scripts/railway/env-sync.sh
    ```

    Railway auto-redeploys when values change.

  </Step>
</Steps>

## Auto-deploys from GitHub

By default every code update needs `./scripts/railway/redeploy.sh`. To auto-deploy on every push to `main`:

1. Open the Railway dashboard → your project → the `agent-os` service → **Settings**.
2. Under **Source**, click **Connect Repo** and pick your repo.
3. Set the deploy branch to `main`.

Push to `main` now triggers a build and rolling deploy. `./scripts/railway/env-sync.sh` is still how you push env changes.

## Opting out of JWT (not recommended)

If you must run production without auth (inside a private VPC behind another auth layer), set `authorization=False` in `app/main.py` and redeploy. Keep authorization on for any deploy holding real data. Without it, anyone who guesses your Railway domain can read your sessions and run your agents.

## Scaling

`railway.json` deploys one replica at 4Gi memory and 2 vCPU. Keep `numReplicas` at 1: the scheduler runs in-process and the Starter is designed to run as a single replica. Multi-replica HA is not supported yet. Raise `limits` as usage grows.

## Operations

| Task | Command |
| ---- | ------- |
| Tail logs | `railway logs --service agent-os` |
| Open the Railway dashboard | `railway open` |
| Push env changes | `./scripts/railway/env-sync.sh` |
| Sync a different env file | `./scripts/railway/env-sync.sh .env` |
| Redeploy without a git push | `./scripts/railway/redeploy.sh` |
| Tear down (deletes all data) | `./scripts/railway/down.sh` |

`down.sh` asks you to type the project name before deleting, then comments out the stale `AGENTOS_URL` in your env file so a future `up.sh` starts clean.
